## 为什么我们需要并行？

现代 CPU 通常都包含多个**核心**（Core），每一个核心在同一时刻只能执行一条**指令**（Instruction）。你之前写过的各种程序，在默认情况下都只运行在一个**线程**（Thread）中，而操作系统通常只会把这个线程**分配到一个 CPU 核心上**执行。这意味着，如果你什么都不做，即使你的电脑上有 8 个、16 个甚至更多的核心，事实上**只有一个核心**在执行你写的程序，其余的处于**空闲状态**。

~~这确实很令人难过，一方面是因为花钱买了电脑却发现只用了它一点点性能；另一方面，有些时候它真的跑的很慢！~~

比如你可能有一个有趣的功能，我们就暂时称它为 `do_something_good()` 吧，你可能真的会想对 1000 个输入进行你有趣的操作？所以出于直觉你一下子就写出了你最擅长的 `for` 循环（相信大部分人应该更喜欢写 `for` 吧），熟练的填上大一就写烂的 `i++`，编译执行！

```c lineNumbers
int sum = 0;

// [!code word:1000]
for (int i = 0; i < 1000; i++) { // [!code highlight]
    sum += do_something_good(i);
}
```

然而大部分时候它看起来也是在瞬间完成的，<Spoiler>就像你在大一的时候写的 C 语言作业一样快</Spoiler>。

毕竟你写的 `do_something_good()` 应该也没有多复杂就是了。你的电脑还是没有那么笨的，但是**并不代表它真的一直都很聪明**。总有一天你发明出了一些比较复杂的函数，比如一个<Spoiler>自动</Spoiler>完成你作业的函数 `do_my_homework()`，比如操作系统每周的作业，<Spoiler>学习通有那么多题</Spoiler>，写一题要 10 秒的话，写 50 题就需要快九分钟！

```c lineNumbers
for (int i = 0; i < 50; i++) {
    do_my_homework(); // 10 秒一次 // [!code highlight]
}
```

如果你还记得一些计算机组成原理的知识的话，<Spoiler>或许你已经忘了</Spoiler>，你应该知道 CPU 的一个核心一个时候只能做一件事情。~~哪怕忘了也没有关系~~，希望你对做实验的那台**上古电脑和仪器**有点印象，从 `PC` **程序计数器**出来的东西一次只有一个，实际上它确实只有一个，所以单核心的 CPU 和人脑一样，它**一次也只能做一道题**。

你买电脑的时候肯定看过吧，至少知道电脑有几个核心，或者起码**知道它肯定不是单核的**。反正你突然觉得如果可以让我的电脑的好几个核心**同时帮我写作业**应该是一件很酷的事情，<Spoiler>但是还是不要去破解学习通哈</Spoiler>。

<Callout title="概念复习 · 并行" type="info">
  把任务分配给**多个核心同时进行处理**，这个过程就叫**并行 (Parallelism)**。
  并行的核心思想，就是把原本必须**顺序执行**的工作拆分成**多个可以同时进行**的部分，让它们在不同的
  CPU 核心上**并发**执行。
</Callout>

<Callout title="并发和并行的区别" type="warn">

并发（Concurrency）关注的是**程序的结构设计**，指的是一个程序能够**同时处理**多个任务，这些任务的完整**周期在时间上重叠**，但**不一定**真的在同一瞬间完成。简单得多，并发是**同时管理很多事情**。

并行（Parallelism）关注的是程序的实际执行，指的是多个任务真的在**同一时刻进行**，是真正意义上的**同时做很多事情**，需要硬件的支持才能真正实现，而**多核心 CPU** 就是这个硬件。

</Callout>

<Callout title="概念复习 · CPU 核心与线程" type="info">

CPU 核心（Core）是硬件层面的**物理执行单元**。每个核心在**同一时刻**只能高效执行一条指令流（单指令流）。多核
CPU 就像有多条**并行的流水线**。

线程（Thread）是**软件层面的执行单元**，代表一段**独立的指令序列**（可以运行的代码）。操作系统负责把**线程调度**到可用的 CPU 核心上去执行。

</Callout>

## 从手动并行到 OpenMP

我们当然可以**手动创建和管理**多个线程（例如使用 `std::thread` 或 `pthread`）。这种方式虽然非常灵活，但需要自己处理线程创建、任务划分、数据同步等问题，**代码复杂容易出错**。我们不希望自己分配，<Spoiler>学会高效偷懒是进步的开始</Spoiler>，创建线程，四五个可能还好，那几十个呢？

<Callout title="延伸：代码生成（Codegen）" type="idea">

面对以上这类的问题，你可能会说，既然写几十个也是**一种机械的任务**，为什么不能让计算机**自己做**呢？

当然，这是当今现代软件开发里非常常见的一种手段，名字叫做**代码生成**（Codegen），即使用一份代码**生成**更多的其他结构性的代码。比如，你的一个程序中，可能有很多地方是**基本重复**的，但是每一份之中都有几个值的名字不一样，这种时候**提取函数不一定**能解决问题，如果可以写一个**生成绝大多数重复代码**的字符串程序，那或许就可以解决。

当然，最适合生成程序的地方毋庸置疑，是**编译器**；你应该已经见过这样的概念了，比如 C 语言的 `#define`，它是一种**编译器指令**，用于替换代码的文本本身，一般称为**宏**（Macro）。

</Callout>

所幸，OpenMP 可以为我们创建这些**处理线程的程序**，它也是一套**编译器指令**（Compiler Directives），通过在代码里加入 `#pragma omp` 开头的指令，编译器会自动完成，**创建数量合适的线程**、**循环迭代分配**并**处理线程同步**。

最基础的用法是在 `for` 循环前添加 `parallel for` 指令：

```c
#pragma omp parallel for // [!code highlight]
for (int i = 0; i < 1000; i++) {
    // 一些比较慢的操作
}
```

此时，编译器会在生成的程序中为你创建并管理线程，把这 1000 次任务**均匀**分给它们；操作系统就会**调度这些线程**到 CPU 核心上去执行。

[Link](/os-2026-spring)

```c
int a = 1; // [!code --]
int b = 1; // [!code ++]
```

## 为什么需要 Reduction ？

看起来很完美对吧？但如果我们直接把刚才求和的代码加上这句魔法：

```c
int sum = 0;

#pragma omp parallel for
for (int i = 0; i < 1000; i++) {
    // [!code word:sum]
    sum += calculate_complex_math(i); // [!code focus]
}
```

**程序会算出一个完全错误的随机数！为什么？**

想象一下：4 个小学生同时算完了一道题，同时冲到黑板前，拿起粉笔要在同一个位置把自己的结果加上去。A 刚擦掉旧数字准备写新结果，B 的手就伸过来盖住了 A 的字……黑板上的数字瞬间变成了一团乱码。

在计算机里，多个线程在同一时刻去读写同一个变量（这里是 `sum`），会导致数据互相覆盖，这被称为**数据竞争 (Data Race)**。

**怎么解决？**

最聪明的做法是：给每个小学生发一张自己的草稿纸！
让 4 个人先在自己的草稿纸上把那 250 道题加完。等所有人都算完了，再依次把 4 张草稿纸上的总数加到黑板上。

OpenMP 把这个过程完美地封装成了一个指令：**`reduction`（归约/合并）**。

```c
#pragma omp parallel for reduction(+:sum)
for (int i = 0; i < 1000; i++) {
    sum += calculate_complex_math(i);
}
```

`reduction(+:sum)` 这短短几个字，告诉了编译器三件事：

1. 帮每个线程克隆一个名为 `sum` 的“草稿纸”（局部变量）。
2. 因为后面跟的是 `+` 号，所以每张草稿纸的初始值设为 `0`（加法的自然起点。如果是乘法 `*`，初始值就是 `1`）。
3. 循环结束后，自动把所有草稿纸上的值按 `+` 号合并，加到最终真实的 `sum` 变量里。

完美解决！

## 为什么还需要调度？

现在求和没问题了，但新的问题又来了。

假设这 1000 道题里，有的是极其复杂的微积分，有的只是简单的 `1+1`。OpenMP 默认的分配方式是 **`static`（静态分配）**：把 1000 题均分成 4 份，每人 250 题。

这会导致什么结果？如果小学生 A 分到的全是微积分，另外三个人分到的全是 `1+1`，那么另外三个人 1 分钟就算完了，剩下的小学生 A 要通宵算到天亮！整个程序的耗时，完全被最慢的那个线程拖垮了。

**怎么解决？**

我们改变分配策略：**`dynamic`（动态分配）**。不提前分好题目了。我们把 1000 题放在一个公共题篓里。谁先做完手上的题，谁就去题篓里再拿一道做。能者多劳！

```c
#pragma omp parallel for schedule(dynamic)
```

**隐藏的性能瓶颈！**
这个“公共题篓”每次只能允许一个人来拿题目，这就成了一个 **临界资源**（就像寝室里只有一个卫生间，大家必须排队洗澡）。
如果 10000 题，4 个人每做完一小题就跑去题篓排队拿下一题，排队浪费的时间甚至比做题的时间还长！

所以，我们让他们每次去拿的时候，直接拿走 **一块 (chunk)** 题目：

```c
#pragma omp parallel for schedule(dynamic, 8)
```

这里的 `8` 就是 chunk 的大小。每次排队直接领 8 题回去慢慢算，大大减少了排队的拥堵。

**`guided`**
`guided` 是一种结合了静态和动态优点的智能分配方式。一开始题篓满满的时候，它一次性给你一大块（比如 100 题），极大地减少排队；当题目快做完的时候，每次给的块越来越小（最后可能只给 1 题），防止有人在最后关头领了一大堆微积分导致其他人全在等他。
也就是：**前期大块分减少排队开销，后期小块分保证绝对的负载均衡。**

## 软硬协同的底层真相：缓存同步与 TLB 刷新

最后，拓展两个极其硬核的系统级知识。为什么我们说多核并行并不总是能带来线性的性能提升？因为在底层，硬件和操作系统都在为了“保证数据不出错”而付出巨大的同步代价。

**1. 硬件层面的代价：缓存一致性 (Cache Coherence)**

为了让 CPU 跑得更快，每个核心都有自己专属的超高速**缓存 (Cache)**。
当核心 A 修改了自己缓存里的变量 `sum` 时，核心 B 缓存里的 `sum` 还是旧数据。如果没有同步机制，核心 B 就会用错数据。
为了保持数据一致，一旦某个核心修改了共享变量，CPU 硬件在底层就必须通过总线疯狂发送信号（比如通过 MESI 协议），去作废或更新其他所有核心里的旧缓存行。这也就是为什么在写多线程程序时，**尽量让不同的线程去操作不同的数据（比如每人一张草稿纸），减少对同一块内存的频繁竞争**，是提升多核性能的终极奥义。

**2. 操作系统层面的代价：TLB 刷新 (TLB Shootdown)**

除了硬件自己同步缓存，操作系统（OS）在管理内存时也有多核同步的烦恼。
我们知道，程序代码里用的都是“虚拟地址”，CPU 每次读写内存都要把虚拟地址转换成“物理地址”。这个转换过程需要查“页表”。为了加速查表，CPU 里面有一个专门用来缓存页表的硬件组件，叫 **TLB (Translation Lookaside Buffer，快表)**。和 Cache 一样，每个 CPU 核心也有自己专属的 TLB。

想象这样一个场景：操作系统决定把某块内存页回收了，或者把它换到了硬盘上。OS 必须立刻去修改页表，宣告这个虚拟地址失效了。
但这个时候，其他 3 个核心的 TLB 里可能还缓存着旧的页表映射！如果它们继续用旧的映射去写内存，就会发生灾难性的越界错误。

为了解决这个致命绝境，OS 会触发一种叫 **TLB Shootdown (TLB 击落)** 的操作：
当一个核心修改了页表，OS 会立刻向其他所有核心发送一种特殊的硬件中断（IPI，核间中断）。其他核心收到信号后，必须立刻停下手中的工作，清空自己 TLB 里的旧缓存，然后再恢复运行。
这个过程极其昂贵，因为它强行打断了其他核心的流水线！所以在多核时代，操作系统设计者们也是绞尽脑汁，想尽一切办法来减少这种跨核心的同步开销。

> 了解了这些，你就会明白：**“并行”绝不是简单地多加几个 CPU 那么简单。** 背后是无数天才工程师在硬件和操作系统层面上做出的精妙妥协。
